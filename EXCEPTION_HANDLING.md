# Exception Handling Implementation: Special Return Value Model

## Overview
This document describes the exception handling implementation in p7, using a **special return value** approach instead of setjmp/longjmp style exception handling.

## Design Philosophy

Instead of using stack unwinding with setjmp/longjmp (SEH-style), p7 implements exceptions as **tagged return values**. This approach:
- Avoids the complexity and overhead of setjmp/longjmp
- Makes exception flow explicit in the bytecode
- Simplifies the VM implementation
- Makes exception handling more predictable and debuggable

## Data Type Model

### Exception as a Data Variant

The `Data` enum has been extended with an `Exception` variant:

```rust
pub enum Data {
    Int(i32),
    Float(f64),
    StructRef(u32),
    Exception(i32),  // Exception value (enum variant ID)
}
```

An `Exception` is just an `Int` value with special semantics - it represents a thrown exception that should propagate up the call stack.

## Bytecode Instructions

### 1. Throw
**Opcode:** 25
**Semantics:** 
- Pops value from stack
- Wraps it as `Data::Exception(value)`
- Pushes exception back on stack
- **Immediately returns from function** (pops stack frame)

This causes the exception to bubble up to the caller.

### 2. CheckException(address)
**Opcode:** 26
**Semantics:**
- Peeks at top of stack (doesn't pop)
- If value is `Data::Exception(_)`, jumps to `address`
- Otherwise, continues normally

Used in `try` expressions to detect if the result is an exception.

### 3. UnwrapException
**Opcode:** 27
**Semantics:**
- Pops top of stack
- If it's `Data::Exception(value)`, pushes `Data::Int(value)`
- Otherwise, raises runtime error

Used to extract the exception value for pattern matching in `else` blocks.

## Code Generation Patterns

### try expr (propagation form)

```p7
fn[throws] foo() -> int {
    let x = try bar();
    x
}
```

Generated bytecode:
```
Call bar
; Result is on stack - could be normal value or Exception
; No CheckException needed - if it's an exception, it will 
; automatically propagate when this function returns
```

### try expr else handler (handling form)

```p7
fn foo() -> int {
    let x = try bar() else 42;
    x
}
```

Generated bytecode:
```
Call bar                    ; Call bar(), result on stack
CheckException L1           ; If result is Exception, jump to L1
Jmp L2                      ; Normal case: skip handler
L1:                         ; Exception handler
  UnwrapException           ; Extract exception value
  Ldi 42                    ; Generate else expression
  ; (else expression can use the unwrapped exception value)
L2:                         ; Continue
  ; Result (either from bar or from else) is on stack
```

### try expr else { pattern matching }

```p7
fn foo() -> int {
    let x = try bar() else {
        err: MyError.Failed => 1,
        err: MyError.Other => 2,
    };
    x
}
```

Generated bytecode:
```
Call bar
CheckException L1
Jmp L_END
L1:
  UnwrapException           ; Exception value now on stack as Int
  ; Pattern matching branches go here
  ; Each Statement::Branch checks the value and executes its expression
  Ldi 1                     ; First branch expression
  Jmp L_END
  Ldi 2                     ; Second branch expression
L_END:
```

## Runtime Behavior

### Throwing an Exception

1. `throw SomeError.Failed` generates:
   - Load enum variant ID
   - `Throw` instruction
2. `Throw` instruction:
   - Pops the value
   - Wraps as `Data::Exception(value)`
   - Pushes back on stack
   - **Pops stack frame and returns**
3. Caller now has `Data::Exception` on stack

### Catching an Exception

1. After calling a throws function, `CheckException` peeks at result
2. If it's `Data::Exception`:
   - Jumps to exception handler
   - Handler uses `UnwrapException` to get the value
   - Handler code executes (may pattern match on value)
3. If it's a normal value:
   - Execution continues normally

### Exception Propagation

If a function doesn't have an `else` handler:
- Exception stays on stack as `Data::Exception`
- When function returns, exception is still on caller's stack
- Caller's `CheckException` will detect it
- This continues up the call stack until handled

If an exception reaches the top-level without a handler:

## VM Implementation Details

### Operation Handling

All arithmetic and comparison operations check for `Data::Exception`:

```rust
match (a, b) {
    (Data::Int(a), Data::Int(b)) => { /* normal operation */ }
    (Data::Exception(_), _) | (_, Data::Exception(_)) => {
        return Err(RuntimeError::Other("Operation on exception value"));
    }
    // ...
}
```

This prevents exceptions from being used in computations accidentally.

### Stack Frame Behavior

When `Throw` executes:
1. Exception wrapped and pushed on stack
2. Stack frame immediately popped
3. Exception is now on **caller's** stack
4. Caller continues execution with exception on stack

This is different from normal `Ret`:
- `Ret`: Pops value, pops frame, pushes value on caller's stack
- `Throw`: Wraps value as exception, pops frame (exception already on caller's stack)

## Advantages of This Approach

1. **Simplicity**: No setjmp/longjmp, no complex unwinding
2. **Explicitness**: Exception flow is visible in bytecode
3. **Performance**: No need to save/restore execution context
4. **Debuggability**: Can trace exception flow through bytecode
5. **Type Safety**: Exceptions are part of the type system (Data enum)
6. **Composability**: Easy to add exception handling to existing code

## Future Enhancements

1. **Pattern Matching**: Full implementation of pattern matching in `Statement::Branch`
2. **Named Patterns**: Binding exception values to variables
3. **Multiple Exception Types**: Support for typed exceptions
4. **Stack Traces**: Collect call stack info when exception is thrown
5. **Uncaught Exception Handling**: Explicit check at entry point

## Comparison with Other Approaches

| Approach | Pros | Cons |
|----------|------|------|
| setjmp/longjmp | Industry standard | Complex, platform-specific, hard to debug |
| Table-based unwinding | Used in C++ | Requires complex metadata, overhead |
| **Special return value** | Simple, explicit, debuggable | Requires runtime checks on every call |
| Sum types (Result<T,E>) | Type-safe, explicit | Verbose, requires monadic handling |

The special return value approach strikes a good balance for p7's goals of being a simple, understandable language with low implementation complexity.
